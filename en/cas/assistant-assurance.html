360 chatbot · Client · Insurance

A chatbot that answers the client, triggering the right process

A policyholder or a prospect turns up with a question, without knowing which form or tool to use. An umbrella assistant detects the intent and triggers the right process: subscribe, understand a policy, file a claim, track a case. A plain-language, reassuring tone for the general public.

The 360 view comes in four cases. This page is the client-side 360 chatbot. It has a twin with a different tone, the 360 chatbot for general agents. Both sit on the 360 context, which gathers everything known about a policyholder, and both draw on the 360 contracts view. The context is not just a chatbot brick: it is the foundation.

The idea

You start by building specialised assistants, each excellent at one task: one for underwriting, one for policy questions, one for filing a claim, one for tracking. Then you place a single agentic assistant on top. It does not do everything itself; it understands the request and hands it to the right specialist. For the user it is one door; behind it, each process stays clean and controlled.

How it works

Understand the requestThe question is analysed for the real intent, not just the words.
Route to the right specialistThe intent triggers the right process: underwriting, policy, claim, tracking.
Let the specialist workEach specialised assistant keeps its own tested logic, with a typed output.
Stay in the loopThe umbrella assistant follows the exchange, re-routes if the intent changes, and keeps the trail.

The 360 client view

Because it sits above every process, the assistant sees the whole client: their policies, their claims, their history. A single conversation can move from a coverage question to a declaration, then to tracking, without the user switching tools. That is the 360 view: one door, every business behind it.

A concrete example of the back-and-forth between bricks: a client starts a home-insurance subscription and asks, "and my bike, if it's stolen outside my home?". The assistant switches to the sourced answer on the policy (off-premises theft covers it, when the household policy provides for it), then to the comparison to show that some competitors do not cover it, and returns to the subscription. One conversation, three bricks.

Technical roadmap: a modular platform, project by project

The point is not to ship five separate products, but to build a single modular AI platform. The core (document parsing, structuring, RAG, dialogue engine, API, conversation handling and traceability) is built once. Each project reuses the previous bricks and adds a layer, which cuts redundant work and speeds up the next use case to market.

Result: the same engine reasons over several policies. See the criterion-by-criterion comparison.

Underwriting assistantInternal API · read + write

Move from "answering" to "guiding" a subscription.

  • Multi-step dialogue, structured collection, data validation.
  • Connection to the information system.
  • Creating or updating records, persistence.

Result: the chatbot triggers business actions. See the subscription chatbot.

Claim-filing assistantInternal APIAuthentication

Reuse the understanding of policies to assist the declaration.

  • Analysing the account of the loss, checking the guarantees.
  • Generating a justification, assembling the file automatically.
  • Recording the exchanges, handing the file to the management apps.

Result: the chatbot guides all the way to the claim file. See the guided declaration and the plain-language tracking.

Intelligent settlementInternal APIDecision + human

Automate the analysis of the file to offer immediate settlement where possible.

  • Analysing supporting documents and images.
  • Fusing information from the policy, the documents and the photos.
  • Decision engine, detection of cases that need a human, integration with the settlement system.

Result: for simple files, settlement can be offered automatically.

The throughline: each project reuses the previous bricks and adds a layer (comparison, workflow orchestration, IS integration, multimodal analysis, decision). The core is built once; the next use cases arrive faster and cost less.

What it shows

A "chatbot that does everything" is not a giant model that guesses: it is an agentic orchestrator on top of solid specialised assistants, sitting on a shared platform. The value is in reliable intent routing, in the quality of each brick, and in building the core only once. The real effort is less about the dialogue than about the dependencies: public documents on one side, information-system plumbing and authentication on the other.
