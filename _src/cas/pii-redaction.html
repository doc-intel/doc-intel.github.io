Redacting personal data from a CV

Before a résumé reaches any model or gets shared, the obvious personal data has to go: emails, phone numbers, names. A deterministic pass catches the easy, high-confidence cases first, cheaply and reproducibly.

The problem

A recruiter's pipeline processes hundreds of résumés. Sending them to a model, or sharing them across a team, means personal data leaves its original context. Much of that data follows regular shapes: an email is an email, a phone number is a phone number. Those should be masked deterministically, before anything more expensive or less predictable runs.

The approach

A regex layer scans the document and flags the personal-data entities it recognizes with high confidence, then masks them in place. It is deterministic (same input, same output), cheap, and auditable: each masked span is a match you can point to. The harder, ambiguous cases (a name that looks like a common word, an address split across lines) are left to a later, model-based pass; the regex layer's job is to remove the easy 80% for free.

The thread: redaction starts before the model. The cheap, deterministic layer removes what has a regular shape, so the expensive layer only sees what actually needs judgement.

The method, step by step

Scan for known shapesEmails, phone numbers and other regularly-shaped entities are matched by pattern.
Mask in placeEach match is replaced, keeping the document readable, every masked span traceable to its match.
Leave the hard casesAmbiguous entities are deferred to a later model-based pass, not forced through regex.
Stay deterministicSame input, same output; no model call for the easy layer.

What it shows

Anonymization is a cascade, not a single call. The first layer is deterministic and free, and it should run before anything reaches a model. What is left for the model is only what genuinely needs judgement.
