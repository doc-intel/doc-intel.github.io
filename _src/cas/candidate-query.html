Querying the candidate pool in plain English

A recruiter asks in plain words; the assistant turns it into a bounded query over the candidate database, runs it, and shows the query as much as the result.

The problem

A recruiter knows the profile they need but not the database schema or SQL. The candidates exist somewhere in a table; getting to the right ones should not require writing queries.

The approach

The assistant knows the candidate table schema and generates a read-only SELECT from the request. The query is bounded (no writes, validated columns), executed, and returned with the SQL in plain sight, so the recruiter sees exactly what was asked.

The thread: the assistant only translates the need into a bounded query. The query is shown, so it is verifiable, not a black box.

What it shows

A candidate pool does not need fuzzy search to be queried: once the data is typed, the recruiter's request drives a deterministic, checkable query.

Deliverables

An interface that takes a recruiter's request, shows the generated SQL and the matching candidates. The candidate data is synthetic and anonymized.
